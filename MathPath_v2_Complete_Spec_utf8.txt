Every design decision must reduce extraneous load, manage intrinsic load, and maximize germane load. Four-Level Difficulty Progression Level Name Description 1 Concrete &amp; Visual GeoGebra-led exploration with real-world context. No independent problem-solving. Fully worked examples only. 2 Procedural (Fading Guidance) Connect visuals to MathJax formulas. Introduce early active recall. Partial worked examples with student completion. 3 Abstract Application Remove visual aids. Require independent multi-step problem solving. Worked examples fully faded. 4 Complex Derivations Mastery level for non-standard heuristics — Diophantine equations, advanced physics derivations, proof construction. Expertise Reversal Trigger (Hardcoded Rule) 📌 When a user scores &gt;80% on practice questions within a module, automatically fade out worked examples and require independent problem-solving. This is a deterministic system rule — no exceptions. Score &lt; 60%: serve fully solved worked examples before every practice question Score 60–80%: serve partially completed worked examples (student fills final steps) Score &gt; 80%: remove all worked examples — independent solving only On module retry (after lockout): reset to full worked examples regardless of prior score Content Module Schema ⚠ Every module must conform to this schema. PART B ADDENDUM — SESSIONS, API, DATABASE, INSTITUTIONAL &amp; SECURITY B13 — SESSION MANAGEMENT (EXTENDED SPEC) 📌 The original B4 covers auth methods and token durations. This section covers all remaining session mechanics, device management, Redis structure, and the offline JWT problem. B13.1 — Token Architecture Token Specification Access token (JWT) 15-minute expiry. Contains: user_id, tier, role, lang_pref, session_id. Signed with RS256 (asymmetric). Public key exposed at /.well-known/jwks.json. Refresh token 30-day expiry. Opaque random string (32 bytes, hex-encoded). Stored in Redis with TTL. httpOnly cookie, SameSite=Strict, Secure flag. Token rotation On every refresh: old refresh token invalidated, new one issued. Redis key updated atomically. Prevents refresh token replay attacks. Refresh token family tracking If a previously-invalidated refresh token is used: entire token family invalidated, all sessions for that user terminated. Indicates token theft. CSRF protection Double-submit cookie pattern for all state-mutating requests. Not needed for JWT-in-header requests from mobile. B13.2 — Redis Session Store Structure Key pattern: session:{user_id}:{session_id} TTL: 30 days (matches refresh token) Value (JSON): { user_id: string, session_id: string (UUID v4), refresh_token: string (hashed — never store raw), device_type: &apos;web&apos; | &apos;ios&apos; | &apos;android&apos; | &apos;pwa&apos;, browser: string, ip_address: string (hashed for privacy), location: { city, country }, created_at: ISO timestamp, last_active: ISO timestamp, tier: &apos;free&apos; | &apos;pro&apos; | &apos;institutional&apos;, is_offline_capable: boolean } Secondary index: sessions_by_user:{user_id} → SET of session_ids Used for: listing all sessions, revoking all sessions for a user Updated atomically with session create/delete B13.3 — Offline JWT Expiry Problem ⚠ This is the most overlooked session issue for offline-first apps. A user goes offline at 9am with a valid JWT. JWT expires at 9:15am. User is still offline at 10am. Without a solution, they are silently logged out and lose offline access. Solution: extended offline token stored separately in service worker cache On successful online auth: issue standard 15-min JWT + an &apos;offline continuation token&apos; (8-hour expiry, signed differently, flagged as offline-only) Offline continuation token: only valid when navigator.onLine === false. Rejected by server if sent online. When online JWT expires while offline: service worker transparently switches to offline continuation token for any cached API calls On reconnection: full re-authentication forced. Offline continuation token invalidated. Fresh JWT issued. Offline continuation token scope: read-only access to cached content + queued assessment submission only. Cannot unlock modules, change tier, or access leaderboard. Assessment sessions started offline: tagged with offline_session=true. Server validates extra carefully on sync.
